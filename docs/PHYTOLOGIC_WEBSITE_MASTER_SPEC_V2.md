## 1. 設計定位

官方網站 = **品牌形象站**，不是商城、不是會員系統。它唯一的任務：
讓第一次接觸植本邏輯的人，在 90 秒內感受到「這是一群真實的人，用真實的土地與邏輯，在守護重要的人」。

設計哲學三角：
- **Minimal Luxury**：留白、節制、不喧嘩
- **Calm Intelligence**：六角邏輯結構、清晰的資訊層級
- **Botanical Science**：大自然景觀的光影敘事 + 科學感的版面節奏

明確排除：生鮮電商感、過度飽和的健康食品廣告色調、誇張醫療宣稱、雜亂拼貼。

## 2. 視覺方向：AI 生成圖像（大自然景觀 + 局部剪影 + 光影效果）

因實際拍攝資源無法到位，全站素材來源為 **AI 生成圖像**。視覺題材方向：**不生成近距離蔬果細節，不生成全臉人物**，全站圖像聚焦在「大自然景觀 + 局部身體剪影 + 光影效果」三個元素的組合。

這個收斂讓六色之間的差異純粹依靠「場景類型＋光影色溫」來區分，而非食材顏色特寫的堆疊——與「minimal luxury」的留白哲學一致，也降低 AI 生成在「近距離細節」（手指、果肉紋理、蔬果表面）上最容易露出「AI感」的風險區域。

### 2.1 題材方向

| 類別 | 規範 |
|---|---|
| 自然景觀（主力題材） | 清晨、森林、高山、田園、果園、農田等大景，作為每個系列的場景基底，依各系列主題選擇對應地景（見第5節各色場景對照） |
| 局部身體剪影 | 僅限背部、手部等局部，逆光或側光剪影處理，**禁止生成完整臉孔**，人物在畫面中應是點綴而非主體焦點 |
| 光影效果 | 每張圖必須有明確的光線敘事（晨光穿林、夕陽染田、雲霧繚繞山頭等），光影是畫面情緒的主要載體 |

### 2.2 明確禁止
- 近距離蔬果/食材特寫（果肉剖面、葉脈紋理、水珠等微距畫面，一律不生成）
- 完整、清晰的人物臉孔（不論正面或側面五官清晰可辨）
- 棚拍式均勻照明、過度置中對稱構圖

### 2.3 AI 生成品質規範

| 項目 | 規範 |
|---|---|
| 光線 | 自然光敘事為核心 prompt 元素（晨霧、黃昏側光、林間透光），避免均勻無方向性的光照 |
| 色調 | 保留地景與光線的自然色階，不過度飽和、不套用單一濾鏡式統一色調 |
| 構圖 | 大量留白／負空間，地景與人物剪影皆採偏置構圖，避免置中對稱 |
| 人物比例 | 人物剪影在畫面中應佔小比例（建議不超過畫面1/4），強調人融入自然的渺小感與寧靜感 |
| 品質檢核 | 上線前人工檢查：剪影肢體比例是否合理變形、光影是否符合物理邏輯、地景元素是否有重複貼圖感的「AI紋理」 |

### 2.4 人物素材原則（硬性規則）

人物相關畫面必須謹慎處理，理由如下：
1. 完整人臉若用 AI 生成，一旦被識破，容易讓人質疑品牌畫面的真實性
2. AI 生成人臉容易出現 uncanny valley（似真又假）的詭異感，破壞「溫暖、可信」的調性

**人物相關畫面僅允許以下處理方式：**
- 逆光剪影（如黃昏中牽手的背影、晨光中田間勞作的剪影）
- 局部身體特寫（手部動作、背影、肩膀依靠），畫面中不出現臉部五官
- 人物為遠景中的渺小存在（如山徑上的小身影），自然成為「無法辨識五官」的安全處理

### 2.5 AI 生成工具建議（已確認）

**主力工具：Adobe Firefly**

選擇理由：
1. **商用安全性**：Firefly 僅使用 Adobe Stock 授權內容與公開領域素材訓練，不使用網路爬取的未授權圖像，Adobe 提供商用智慧財產權侵權賠償保證（indemnification）。植本邏輯主打「無人工、無合成」的品牌信任，圖像來源的合法性與可追溯性本身就是品牌一致性的一部分，這點上 Firefly 是目前市場上最乾淨的選擇
2. **光影與構圖控制力**：Firefly Image Model 4 對「精確控制光線與鏡頭角度」有專門優化，正好對應本規範第2節「自然光敘事為核心 prompt 元素」的需求
3. **品牌一致性**：Firefly 支援 Custom Models（可用品牌素材微調），若六色頁面需要長期維護一致的視覺語言，這個功能比每次重新調 prompt 更穩定
4. **整合性**：與 Photoshop/Illustrator 整合，方便 Claude Design 生成後直接進行二次微調（去背、構圖裁切、剪影強化等）

**輔助參考：Midjourney（僅供 mood board／方向探索，不用於正式商用素材）**

Midjourney 在自然風景、光影氛圍的藝術表現力上仍優於 Firefly，適合在前期用來探索「晨霧山林」「黃昏花田」這類情緒參考圖。但 Midjourney 的訓練資料來源較不透明，商用版權風險高於 Firefly，**建議僅作為內部方向溝通的參考圖，正式上線的素材仍以 Firefly 生成為準**。

**不建議：完全依賴免費／開源工具（如本地部署的 Stable Diffusion）**

雖然零成本且可完全客製，但需要技術團隊投入大量時間調校模型與規避版權風險，對六色頁面需要的「穩定一致性」與「上線時程」而言，效益不如直接使用 Firefly 的商用方案。若未來素材量大幅增加、需要更精細的客製化控制，可再評估導入。

## 4. 首頁結構規範（Section by Section）

### Section 1 — Hero
- 滿版 AI 生成大景（建議：台灣高山或田野晨景，雲霧與晨光，留白偏置構圖；可含遠景中渺小的人物剪影，不可有清晰人臉）
- 疊加文字：品牌核心句，建議取自公司簡介：「對於幸福，重新理解之後的答案：唯有健康。」
- 無多餘 CTA 按鈕堆疊，僅一個輕量向下捲動提示
- 動態：圖片極緩慢的 Ken Burns 式微縮放（呼應「呼吸感」），文字淡入

### Section 2 — 六大植萃系列入口
- 橫向或網格排列六張 AI 生成自然景觀圖（**非食材特寫，是各系列對應的場景**，見第5節各色場景對照）
- **排列邏輯（已確認）：六色平行並列，等比例呈現，鉑金植萃不另做視覺凸顯或置中區隔**——雖然鉑金植萃在產品定位上是「共同基礎」，但在首頁入口的視覺層級上與其他五色一致，差異只表現在各自的故事頁內容
- 每張圖配對一個 Soul Color 細線標記 + 一句核心句（如「能吃就是福，吃的下才可以活得好」）
- 點擊進入各系列故事頁（V1 範圍內可先做卡片懸停預覽，故事頁留待 Phase 2 開發）

### Section 3 — 三好三無原則
- 簡潔的雙欄對照版面：「三好」與「三無」
- 不使用 icon 插畫化呈現，改用極簡文字編排 + 細線分隔，維持邏輯感而非條列感
- 背景可置入低飽和度、失焦模糊處理的自然景觀局部（如遠景山林的模糊光斑）作為紋理，**不可使用食材或植物的近距離特寫**，僅作質感背景

### Section 4 — 信念主張
- 大字級短句呈現品牌相信的事：「當你愛的人還在時，你也能好好地在、能陪伴、能擁抱...」
- 全頁留白，搭配一張極簡的自然光景中牽手背影剪影（避免清晰人臉與擺拍感過重的素材）

### Section 5 — 導流（CTA）
- 唯一的商業轉換點：導向 LINE OA @phytologic
- 文案克制，不使用「立即購買」「限時優惠」等促銷語言，符合品牌「不販賣焦慮」的主張
- 建議文案方向：「想更了解植本邏輯，從這裡開始」

### 頁尾
- 六角形 Logo + 簡短品牌標語
- 基本資訊（公司/聯絡/社群連結）
- 神聖金可在此處作為極細的分隔線或 Logo 描邊色使用

## 5. 六大植萃系列故事頁規範

### 5.0 最重要的一條規則：內容來源邊界

六份《內部機密醫療實驗室報告書》是 R&D / QA 內部文件，文件本身明確標註「不得外流、轉載或直接作為公開廣告宣稱」。因此故事頁內容來源做出**強制區分**：

| 可以使用 | 不可使用 |
|---|---|
| 公司簡介中創辦人/家庭的真實情感敘事（六色故事原文，含鉑金植萃第六個家庭的心聲） | 實驗室報告中的分子機制、通路名稱（SIRT1、NF-κB、GLP-1...） |
| 食材的產地/外觀/風味/傳統食療印象的**感性描述** | 任何百分比數據、劑量數字、臨床研究措辭（「臨床證實」「研究顯示提升 X%」） |
| 中醫五氣五味的**氛圍性**描述（溫潤、清涼，作為風味語言而非療效語言） | 疾病名稱（阿茲海默症、高血壓、腸躁症、甲狀腺機能低下等任何病理名詞） |
| 製作工藝中**可公開的故事性**環節（如「保留種子才能釋放香氣」的工藝講究） | 製程的功能性宣稱（「释放率暴增550%」「生物利用度提升23倍」等技術指標） |
| 三好三無原則的精神性主張 | 「預防」「治療」「降低風險」「改善病症」等醫療化動詞 |

**這條邊界沒有例外。** Claude Design 與任何後續文案產出，若需要引用食材功效相關描述，一律回頭請示本文件的架構權威（Claude／Bryan 核定），不可自行從報告書摘錄改寫後上線。

### 5.1 六頁共用版面模板

每個故事頁結構一致，只置換主色、圖像素材與文字內容，維持品牌的「邏輯與秩序感」：

```
[Section A] 開場全幅照
  └ AI 生成自然景觀大景（單色主導留白；對應該系列的場景基底） + 系列名稱 + 核心句

[Section B] 第○個家庭的心聲（敘事正文）
  └ 該系列對應的家庭故事，以第一人稱書信感呈現研發初衷（六色心聲皆已由 Bryan 確認或提供）
  └ 配圖：局部身體剪影或自然光景情境圖（不出現清晰人臉，不使用食材特寫），呼應情感氛圍而非成分

[Section C] 風土與場景
  └ 介紹該系列食材生長的風土環境（產地印象、節氣氛圍的感性語言，非成分清單）
  └ 圖像：對應地景的 AI 生成大景（如菜園、果園、山林），不使用食材近距離特寫，延續第2節「大自然景觀+局部剪影+光影」標準

[Section D] 三好三無 在此系列中的體現
  └ 簡短呼應品牌原則，不需要每頁重複完整論述，挑與該系列最相關的一兩點即可
  └ 範例：青檸植萃可呼應「好吸收」；玫瑰植萃可呼應「好看」

[Section E] 回到六色入口
  └ 輕量的下一頁/返回首頁導引，維持六色之間的「漫遊感」而非單向跳出
```

## 6. 新增功能（V2.1 追加）：合作聯繫頁 + 內容專區

這是官網第一次出現「靜態頁面」以外的需求，兩項功能性質不同，分開規範。

### 6.1 合作/廠商聯繫頁

**決策（已確認）**：採直接留聯絡資訊，不做表單，零後端，先求快速上線。

- 頁面定位：給有興趣合作的廠商/異業/媒體一個明確的接觸窗口，**不是客服頁面**，語氣延續品牌克制調性，不使用「歡迎洽詢」「業務專線」這類銷售感字眼
- 內容結構：
  - 簡短說明文字（一兩句，說明品牌歡迎什麼類型的合作，如：原料供應商、通路、異業聯名、媒體採訪）
  - 直接列出聯絡信箱與電話（不做表單欄位）
  - 可依合作類型簡單分類聯絡信箱（如：合作洽詢用一個信箱，媒體聯絡用另一個，視 Bryan 是否需要區分）
- 圖像：延續全站「自然景觀+局部剪影+光影」規範，建議用一張安靜、開放感的田野/山林大景作為頁面背景，不需要另外設計視覺語言
- 導覽：建議放在頁尾固定連結，首頁主導覽列視情況決定是否需要獨立入口（不強制）
- **此頁面零後端依賴，可直接交付 Codex 寫成靜態頁面**

> 待確認：聯絡信箱與電話號碼由 Bryan 提供，本規範不可自行假設或填入佔位聯絡方式上線。

### 6.2 內容專區（植本誌）：影片 / 活動花絮 / 健康文章

**決策（已確認）**：需要後台 CMS，讓 Bryan／內容團隊自行上稿，不需要每次發布都找 Codex 改程式碼。

**這是官網第一次需要資料庫與後端**，必須謹慎處理架構邊界：

#### 6.2.1 內容邊界（最重要，三種內容類型風險不同）

| 內容類型 | 風險等級 | 規則 |
|---|---|---|
| 影片 | 低 | 嵌入 YouTube/Vimeo 連結或站內影片，文字描述延續品牌語氣即可 |
| 活動花絮 | 低 | 圖文紀錄，人物入鏡的規則需要另外考慮（見下方說明），無醫療內容風險 |
| 健康文章 | **高** | **必須完整套用本文件第 5.0 節的內容邊界**——不可引用內部實驗室報告的機制數據、不可使用疾病名稱、不可做功效宣稱、不可使用「預防」「治療」「降低風險」等醫療化動詞。健康文章本質上是本規範目前風險最高的內容類型，因為它是「持續產出的新內容」，不像六色故事頁是一次性定稿，每一篇新文章都有重新踩線的風險 |

**活動花絮的人物入鏡規則（與故事頁不同，需單獨說明）**：故事頁規範要求人物不露臉，但活動花絮本質是「真實活動紀錄」，如果是六個家庭或員工自願入鏡的真實活動照片，可露臉；但若使用 AI 生成圖像填充花絮版面，仍須遵守「不生成清晰人臉」的規則。換句話說：**真實活動照可以有臉，AI 生成的配圖不可以有臉**——這條規則由內容上稿者自行判斷使用的是哪一種素材。

#### 6.2.2 CMS 後台規格

- **發布流程**：每篇內容需經過「草稿 → 待審核 → 已發布」三個狀態，**健康文章類型在「待審核」狀態時，須由架構端（Claude）或 Bryan 指定的審核人核對是否符合第5.0節邊界，才能轉為「已發布」**；影片與活動花絮風險低，可由內容團隊直接發布，不強制審核關卡

  **審核權限（已確認）**：Bryan 為最高權限，本人可直接審核，亦可授權其他人擔任審核人。CMS 後台需支援審核角色指定功能（不可寫死成單一人員），讓 Bryan 可隨時新增/變更被授權審核的人員，與既有會員系統「總經理可授權三位領域管理者」的權限邏輯一致，不需另立一套規則。
- **欄位需求**：標題、分類（影片/活動花絮/健康文章）、封面圖、內文/影片連結、發布日期、狀態（草稿/待審核/已發布）
- **前台呈現**：
  - 列表頁：可依分類篩選，卡片式呈現（封面圖+標題+分類標籤+日期）
  - 詳情頁：單篇內容完整呈現，沿用全站字體與留白規範

#### 6.2.3 架構邊界（給 Codex，重要）

- **此 CMS 需要獨立的資料庫，不可共用 PhytoWorld 會員系統的 Supabase 專案**——這延續本文件「Website Design System 與會員系統脫鉤」的同一邏輯，後端也必須脫鉤，避免官網內容操作意外動到會員系統資料
- 若使用 Vercel Serverless Functions，需注意現有專案已有 Hobby Plan 12 個函式上限的問題（會員系統已用 `api/promoter.js` 等資源路由模式緩解），官網若共用同一個 Vercel 專案部署，需採用同樣的單一函式＋resource路由模式，不可每個內容操作都開一個新函式
- 後台登入權限：僅限 Bryan／指定內容團隊成員，不對外開放註冊，避免與會員系統的會員帳號混淆（兩套系統的「使用者」完全不同概念：一個是內容編輯者，一個是消費者會員）

#### 6.2.4 導覽異動

- 首頁主導覽列新增「植本誌」或「最新消息」入口（命名請 Bryan 確認），連結至內容專區列表頁
- 內容專區屬於官網的一部分，但風格與資訊密度可以比六色故事頁稍高（畢竟是持續更新的資訊型內容），不需要嚴格套用故事頁的五段式版面模板，但色彩與字體 tokens 仍須沿用 PHYTOLOGIC Website Design System

> 待確認：CMS 是否要自行開發最小可用後台，或採用現成的輕量級 Headless CMS 服務（如 Notion API、其他第三方工具）來換取更快上線速度，待與 Codex 評估技術選型後再定案。

## 7. 給 Codex / Claude Design 的執行邊界

- 本規範僅定義**內容結構、視覺語言、設計 tokens**，不涉及任何元件程式碼或路由邏輯決策
- Codex 不可自行決定：色值、字體選擇、Section 順序、文案內容 — 任何調整需回到本文件做 Spec Patch
- Claude Design 執行六大故事頁時，需保持版面模板一致（Section A-E 順序固定），差異只在內容與主色，確保使用者跨頁瀏覽時有「同一套邏輯系統」的秩序感
- 文案初稿如需要從食材延伸出感性描述，請先提交文字稿經架構端（Claude）核對是否誤觸第5.0節邊界，再進入視覺執行
- 圖片資源：採 AI 生成圖像（詳見第2節規範），V1 開發階段可先用色塊/灰階佔位框 + 標註所需題材類型，待生成圖通過品質檢核後替換；正式上線前每張生成圖須經人工檢核（剪影肢體/光影/AI感紋理），檢核標準見第2.3節表格
- 故事頁圖像應比首頁更聚焦在「單一系列對應的單一場景與色溫」，避免六頁素材互相混用造成色彩錨點模糊
- 響應式：手機版需重新檢視留白比例（不可等比例縮小桌面留白），但「負空間優先於資訊密度」原則不可妥協

### 7.1 跨頁一致性核對（強制步驟，交付 Codex 前必做）

**問題背景**：六大故事頁與首頁雖共用同一套 PHYTOLOGIC Website Design System，但每頁是獨立生成的 Claude Design session，Design System 只鎖定色票/字體/間距等 token 層級，**不保證圖像選擇邏輯、元件實作細節、文案安全程度跨頁一致**。已發生案例：玫瑰植萃 Hero 圖誤用工具特寫、桂香植萃風土場景誤用蔬果市場特寫——這兩者都不是 token 出錯，而是同一套規則在不同生成中產生的判斷漂移。

**因此，全部七個頁面（首頁+六色故事頁）都完成初版後，必須執行一次跨頁核對，逐項檢查：**

1. **Header / Footer 是否逐字一致**：七頁的導覽列與頁尾結構、文字、連結需完全相同，不可七份各自生成出視覺相似但程式碼不同的版本
2. **間距與版面變數**：每頁 Section 之間的 padding/margin 是否都使用 Design System 定義的 `--space-section` 等變數，而非生成時手寫的新數值
3. **元件樣式**：按鈕、連結、卡片的 hover/active 效果是否跨頁一致
4. **圖像規則複查**：回頭檢查每一頁是否仍符合第2節「不蔬果特寫／不工具特寫／不清晰人臉」三條硬性限制，不只檢查新生成的頁面，已完成的頁面也要重新核對一次
5. **文案安全程度**：六頁的 Section C／D 文案是否都達到紫莓植萃那頁的安全標準（純感性語言、無功效暗示），雪山植萃已知有一句需要留意（「抵達身體需要的地方」），需在此次核對中一併處理

此核對工作建議由架構端（Claude）執行，逐頁截圖或讀取程式碼後列出差異清單，再回頭個別修正，而非交付 Codex 後才發現不一致。
