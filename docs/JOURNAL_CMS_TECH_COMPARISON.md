# 植本誌 CMS 技術選型比較
## 供 Bryan 決策參考

文件性質：技術選型建議（非架構規範，不需版本號）
背景說明：V2 第6.2節規範要求植本誌需要獨立 CMS 後台與獨立資料庫，不可共用 PhytoWorld 會員系統的 Supabase 專案。本文件比較兩個可行方案的優劣，請 Bryan 拍板後再進入開發。

---

## 兩個方案

**方案 A：自建最小可用後台（Self-hosted）**
用 Supabase（獨立新專案）+ Vercel Serverless Functions 自行實作後台介面，Bryan 與內容團隊透過自建的簡單網頁後台上稿。

**方案 B：現成輕量 Headless CMS**
使用第三方 Headless CMS 服務（例如 Sanity、Contentful Free Tier、或 Notion API 作為資料源），官網前台透過 API 取得內容並渲染。後台直接使用該服務提供的編輯介面，不需自行開發後台。

---

## 四面向比較

### 1. 開發時間

| | 方案 A（自建） | 方案 B（Headless CMS） |
|---|---|---|
| 估算 | 2–3 週（後台 CRUD 介面 + API + 權限邏輯 + 審核流程） | 3–5 天（串接 API + 前台渲染，後台直接使用現成服務） |
| 瓶頸 | 後台 UI 本身就是一個小型 Web App，需要登入、角色權限、草稿/審核狀態切換，工作量不小 | 主要工作在前台串接，後台零開發，上線最快 |
| 備注 | 若之後需要調整後台邏輯，每次都要改程式碼 | 第三方服務的後台功能調整不需改程式碼 |

**結論：方案 B 快 4–6 倍。**

---

### 2. 後續內容編輯的易用性（誰會負責上稿）

| | 方案 A（自建） | 方案 B（Headless CMS） |
|---|---|---|
| 上稿介面 | 自行設計，可依需求客製，但初版功能有限 | Sanity / Contentful 等有成熟的富文字編輯器、圖片上傳、預覽功能，接近 WordPress 的使用體驗 |
| 學習成本 | 低（介面由我們設計，可做到最簡單），但功能也較陽春 | 低至中（現成介面已有說明文件，大多數人 30 分鐘內可上手） |
| 健康文章審核流程 | 可完整實作「草稿→待審核→已發布」三狀態，含審核人角色指定 | 大多數 Headless CMS 支援草稿/發布兩態，部分支援自訂審核流程（Sanity 支援，Contentful Free Tier 較有限） |
| **關鍵問題** | 誰負責上稿決定了這項評分的重量——如果是非技術背景的內容人員，方案 B 的現成介面通常體驗更好 | |

**結論：若上稿者為非技術背景，方案 B 體驗較佳；若可接受簡單陽春的自建介面，方案 A 亦可行。**

---

### 3. 獨立資料庫的維運複雜度

| | 方案 A（自建） | 方案 B（Headless CMS） |
|---|---|---|
| 資料庫 | 需開一個全新的 Supabase 專案（與會員系統完全分開），符合 V2 規範的「獨立資料庫」要求 | 資料存放在第三方 CMS 服務的雲端，不在 Supabase，天然與會員系統隔離，亦符合規範精神 |
| 維運責任 | 自行維護：備份策略、資料庫版本、連線安全均需自行處理 | 外包給第三方服務：備份、安全、升級由服務商負責，Phytologic 只需管帳號 |
| 成本 | Supabase Free Tier 足夠初期使用，但若流量增加可能需要付費升級 | Sanity Free Tier：每月 10 GB 資料傳輸、3 位使用者免費，初期完全免費；若未來需要更多使用者或頻寬，約 15 USD/月起 |
| 風險 | 維運責任在 Phytologic 端，若 Codex 日後沒空處理，問題會卡住 | 服務商停止服務或改變定價時，有資料搬遷風險（小但存在） |

**結論：方案 B 維運複雜度低，適合沒有專職技術維運人員的小型團隊。**

---

### 4. 未來擴充彈性（若想加更多內容類型）

| | 方案 A（自建） | 方案 B（Headless CMS） |
|---|---|---|
| 新增內容類型 | 需改資料庫 Schema + 後台 UI + API，每次都要動程式碼 | 在 Headless CMS 後台新增「內容模型」即可，通常不需改程式碼（或只需改前台渲染） |
| 多語言支援 | 需自行實作 | 大多數 Headless CMS 內建多語言支援 |
| 圖片 CDN | 需另外串接（如 Cloudinary），或直接存 Supabase Storage | Sanity / Contentful 內建圖片 CDN 與轉檔功能 |

**結論：方案 B 擴充彈性優於方案 A，且不需要每次改功能都找工程師。**

---

## 建議推薦：方案 B（Headless CMS），首選 Sanity

**理由：**
1. 開發時間最短，可最快讓植本誌上線
2. 維運責任最低，適合 Phytologic 現階段沒有專職技術人員的狀況
3. Sanity Free Tier 初期完全免費，且審核流程（草稿/待審核/發布）支援完整，符合 V2 第6.2節規範中健康文章需要審核才能發布的要求
4. 擴充彈性好，未來若需要加活動頁、產品頁等新內容類型，不需重構

**若選擇方案 B 需要額外確認的事項：**
- Sanity 帳號由誰持有（建議用公司 Email 而非個人帳號）
- 審核人員指定由 Bryan 決定（與會員系統「總經理可授權三位領域管理者」的邏輯一致）
- 前台渲染方式：Sanity 提供 JavaScript SDK，可直接在現有 Vite + React 專案中串接，不需另起新專案

**若選擇方案 A 的情境：** 若 Phytologic 有高度客製後台的需求（例如需要非常特定的上稿流程、或不希望資料存放在第三方服務），方案 A 仍是可行選項，但需要接受 2–3 週的額外開發時間。
